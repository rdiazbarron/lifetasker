One module. One DB call. Four services inject the resolved id.

Problem. The userId() method is copied verbatim into four modules; every read operation triggers a DB upsert — the seam is leaking implementation across every module boundary.

Solution. Extract a UserContextModule with one interface; inject the resolved userId via NestJS request scope rather than resolving it in each service.

  • ✓ locality — user-resolution bugs in one module
  • ✓ leverage — one DB call serves four services per request
  • ✓ interface shrinks on all four services
  • ✓ enables real auth: swap guard, nothing else changes

Collapse the progress calculation into a deep module

apps/api/src/progress/progress.service.ts

Before — shallow (interface ≈ implementation)

interface: currentWeek()
fetch plan + completions
aggregate by blockType
aggregate by category
level: if 15 / if 9 / if 4
percentage = comp / target

Implementation leaks through the interface. 80 LoC, untestable in parts.

After — deep module

interface: currentWeek(userId)
// internal
fetchRaw(userId)
aggregateByType(rows)
aggregateByCategory(rows)
LevelCalculator.compute(n)

Thin interface. Fat implementation. Each piece testable in isolation.

Problem. Level thresholds, category aggregation, and percentage calculation are all inlined in one 80-line method — the module is shallow, its interface nearly matches the full implementation.

Solution. Extract LevelCalculator as a private utility and split aggregation into named private methods; the public interface stays exactly one method.

  • ✓ locality — level bugs isolated to LevelCalculator
  • ✓ leverage — formula testable with plain inputs, no DB
  • ✓ magic numbers become named constants
  • ✓ category aggregation independently verifiable

Solution. Extract a useQuery(fetcher) hook that owns the state machine; pages declare what to fetch, not how.

  • ✓ locality — loading/error bugs in one hook
  • ✓ leverage — 4 pages share one interface
  • ✓ interface shrinks on every page component
  • ✓ caching addable at one seam, not four

Sever the BlockInstances→WeeklyPlans coupling by side effect

apps/api/src/block-instances/block-instances.service.ts
apps/api/src/weekly-plans/weekly-plans.service.ts

Before — hidden coupling

sequenceDiagram
  participant C as Controller
  participant BI as BlockInstancesService
  participant WP as WeeklyPlansService
  participant DB as Database

  C->>BI: complete(dto)
  BI->>WP: getCurrent()
  Note over BI,WP: side effect only — return value discarded
  WP->>DB: upsert plan
  WP-->>BI: plan (ignored)
  BI->>DB: create block instance
  BI-->>C: result

After — explicit seam

sequenceDiagram
  participant C as Controller
  participant BI as BlockInstancesService
  participant PS as PlanBootstrapper
  participant DB as Database

  C->>BI: complete(dto)
  BI->>PS: ensurePlanExists(userId, week)
  PS->>DB: upsert plan
  PS-->>BI: void
  BI->>DB: create block instance
  BI-->>C: result

Problem. BlockInstancesService.complete() calls WeeklyPlansService.getCurrent() purely for its side effect — the return value is discarded, making the seam invisible and untestable.

Solution. Extract plan bootstrapping to a PlanBootstrapper with a narrow ensurePlanExists(userId, week) interface; the intent is explicit at the call site.

  • ✓ locality — plan auto-creation in one module
  • ✓ seam is explicit and testable in isolation
  • ✓ WeeklyPlansModule no longer exported to BlockInstancesModule

Deepen the week-boundary module

apps/api/src/common/week.ts
apps/api/src/weekly-plans/weekly-plans.service.ts
apps/api/src/block-instances/block-instances.service.ts
apps/api/src/progress/progress.service.ts

Before — utility with no seam

WeeklyPlans
BlockInstances
Progress
getWeekBounds() — 14 LoC util

Three direct imports. No seam. Locale/caching behavior must change in all three.

After — one interface

WeeklyPlans
BlockInstances
Progress
WeekService.currentBounds()

Three seams. One interface. Locale or caching changes happen once.

Problem. getWeekBounds() is a bare utility function imported by three modules — it has a shallow interface with no seam, making locale support, caching, or test substitution impossible without touching all three call sites.

Solution. Promote to a WeekService with one injectable interface; consumers depend on the seam, not the raw date computation.

  • ✓ leverage — one module to change week logic
  • ✓ locality — date/timezone bugs in one place
  • ✓ interface stable across 3 call sites
  • ✓ injectable: test adapter substitutable at the seam

Top recommendation

Deepen the user-context seam

It's the root cause behind the most friction: four duplicate implementations, a DB upsert on every read, and the hardest coupling to untangle when auth eventually arrives. Fix this seam first — the other modules immediately become shallower, and the path to real authentication stays open.
